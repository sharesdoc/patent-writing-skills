# 软件与 AI 领域特化防御机制

本文档定义针对软件和人工智能专利的**特化审查规则**，防止交底书被专利局以"纯智力活动规则"、"缺乏技术特征"或"公开不充分"为由直接驳回。

---

## 一、算法的"技术化"包装（反专利法第25条驳回）

### 核心痛点
专利局不保护纯数学公式或抽象的算法改进。

### 防御规则
**严禁直接描述"通过公式 Y=f(X) 提高了准确率"。** 必须强行将算法与具体的物理/数字化技术场景绑定。

### 错误 vs 正确示例

| 维度 | 错误写法 | 正确写法 |
|------|---------|---------|
| 注意力机制 | 改进了 Transformer 的 Attention 机制，降低了计算复杂度。 | 一种**大语言模型文本处理方法及装置**，通过在 Attention 机制中引入稀疏矩阵，**降低了服务器内存占用并提升了文本生成响应速度**。 |
| 损失函数 | Loss=A+B，利用梯度下降求解。 | 将采集到的**车辆图像特征矩阵**输入预设的损失函数进行计算，得到**车辆边缘偏置误差值**，并基于所述误差值调整**卷积神经网络的权重参数**。 |
| 特征融合 | 用公式 X=f(A,B) 融合多模态特征。 | 在所述多模态特征融合引擎301中，将**文本模态的语义向量**与**图像模态的视觉特征矩阵**进行跨模态注意力加权拼接，生成**多维融合特征张量**。 |

### 审查脚本
在 Stage 5 合规审查中执行以下检查：
1. 扫描文本中的独立公式块（`$$...$$` 或 `$...$`）
2. 检查每个公式前后是否有"输入数据来源"和"输出数据用途"的物理场景描述
3. 若公式前后无物理场景描述 → 标记为 🔴 驳回风险

## 二、数据流与控制流的"具象化"（反公开不充分）

### 核心痛点
软件专利最怕"画大饼"——只写概念没有实现路径。审查员会以"无法实现"为由驳回。

### 防御规则
**必须交代清楚以下内容：**
- 输入数据的维度与特征
- 模型的具体层结构（Layer）
- 损失函数（Loss Function）的改良点
- 训练/推理的具体控制逻辑

### 错误 vs 正确示例

| 错误写法（魔法黑盒） | 正确写法（充分公开） |
|-------------------|-------------------|
| 输入图片，通过一个神奇的 AI 模型，就输出了完美的标签。 | 将分辨率为 224×224 的 RGB 图像输入基于 ResNet-50 骨干网络的卷积层进行特征提取，得到 2048 维特征向量；将该特征向量输入由 3 层全连接层构成的分类头，利用交叉熵损失函数计算分类误差，并通过 Adam 优化器反向传播更新网络权重，直至损失函数收敛。 |

### 至少需要交代的内容清单
- [ ] 模型名称或骨干网络类型（如 ResNet-50、ViT-B/16）
- [ ] 输入数据的格式、维度、预处理方式
- [ ] 模型的关键层结构与连接关系
- [ ] 损失函数的数学形式及物理含义
- [ ] 训练/推理的超参数范围（作为示例而非限制）
- [ ] 输出数据的格式与后处理方式

## 三、适度上位与等同替换（扩充保护范围）

### 核心痛点
工程师写"用 Python 编写，部署在 PyTorch 框架下"——抄袭者换成 C++ 和 TensorFlow 就绕过了专利。

### 防御规则
当发明人输入具体实现时，自动将其转化为上位概念。

### 上位概念转换表

| 下位概念（禁止出现在独立权利要求） | 上位概念（独立权利要求中使用） |
|----------------------------------|------------------------------|
| Python / C++ / Java | 计算机可执行指令 / 编程语言 |
| PyTorch / TensorFlow / JAX | 深度学习计算平台 |
| RTX 4090 / A100 / H100 | 图形化处理计算节点 / 并行计算单元 |
| MySQL / PostgreSQL / MongoDB | 关系型数据库 / 非关系型数据库 |
| REST API / gRPC | 网络通信协议 |
| Docker / Kubernetes | 容器化部署平台 |
| 相机 / 摄像头 | 图像采集模块 |
| 激光雷达 / 毫米波雷达 | 距离传感采集装置 |
| BERT / GPT-4 / LLaMA | 预训练语言模型（具体型号仅出现在实施例） |
| GPU / TPU | 硬件加速处理器 |

### 上位与下位结合策略

**独立权利要求（上位，保护范围最大化）**：
> 将数据存储在**关系型数据库**中，通过**网络协议**发送给**远端计算节点**。

**从属权利要求（中位，增加技术细节）**：
> 根据权利要求1所述的方法，其特征在于，所述关系型数据库为 MySQL 或 PostgreSQL。

**实施例（下位，充分公开）**：
> 作为一种优选方案，所述关系型数据库可以为 MySQL 8.0，所述远端计算节点可以为搭载 NVIDIA A100 GPU 的服务器节点，通过 HTTP/2 协议进行数据传输。

## 四、三位一体闭环映射（反逻辑断层）

### 核心痛点
前面提了问题A，后面用方案B解决了问题D——逻辑断层导致创造性被质疑。

### 防御规则
**必须形成严格的三位一体映射：**
```
背景技术痛点(i) ←→ 技术方案特征(i) ←→ 有益效果(i)
```

### 映射表模板
在 Stage 2 方案泛化阶段，必须产出此映射表：

| 编号 | 背景技术痛点 | 本发明技术特征 | 有益效果 | 效果量化指标 |
|------|------------|-------------|---------|------------|
| 1 | 多模态特征融合时信息丢失 | 跨模态注意力加权拼接机制 | 保留模态间语义对应关系 | 多模态检索 mAP 提升 X% |
| 2 | 推理时显存峰值过高 | 稀疏注意力矩阵 + 分级缓存 | 降低显存峰值占用 | 显存峰值降低 Y% |
| 3 | 暗光环境目标检测精度低 | 自适应光照增强预处理模块 | 提升暗光检测准确率 | 暗光 mAP 提升 Z% |

## 五、审查意见答辩防御（反最终驳回）

### 常见驳回理由及答辩策略

| 驳回理由 | 法律依据 | 答辩策略 |
|---------|---------|---------|
| 属于智力活动规则 | 专利法第25条 | 强调算法与具体技术场景的绑定，补充硬件载体或物理世界交互 |
| 缺乏技术特征 | 专利法第2条 | 补充系统架构中的具体硬件模块、数据流向、信号控制逻辑 |
| 公开不充分 | 专利法第26条第3款 | 补充模型内部结构、具体参数范围、训练数据说明 |
| 缺乏创造性 | 专利法第22条第3款 | 强调区别技术特征与现有技术的对比，提供对比实验数据 |
| 权利要求不清楚 | 专利法第26条第4款 | 统一术语、补充定义、修正指代不一致 |
| 保护范围过宽 | 专利法第26条第4款 | 在从属权利要求中增加技术特征限制，保持独立权利要求的上位性 |

## 六、AI 领域特殊红线

### 大模型专利特别注意
- **训练数据说明**：不需要公开具体数据集，但需要说明数据类型和来源范围
- **模型结构公开**：需要公开到层级别（如"包含12层Transformer编码器"），但具体维度可作为实施例
- **提示词（Prompt）不能被保护**：纯提示词设计不是技术方案，必须与模型内部处理逻辑结合

### 计算机视觉专利特别注意
- **数据预处理流程必须写清楚**：归一化方式、数据增强策略
- **后处理不能是简单规则**：NMS、阈值过滤等常规操作不能作为独立发明点

### 多模态专利特别注意
- **融合机制是核心**：必须详细说明不同模态的对齐方式（时空对齐、语义对齐）
- **跨模态注意力权重计算**：必须给出具体的计算逻辑步骤
